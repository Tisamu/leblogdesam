Introduction#

Bienvenue dans le monde de la virtualisation d’environnement, là où tout est possible, où on peux faire n’importe quoi, n’importe où et n’importe quand ! ✨

Plus sérieusement, si vous n’êtes pas familier avec ce qu’est un environnement virtuel aka une VM (Virtual Machine), cet article pourra vous aider à y voir plus clair. 👍

Aujourd’hui les environnements virtuels sont partout, si vous prenez un VPS, c’est littéralement une VM instancié sur un serveur physique qui vous est dédié.

Lorsque vous utilisez des outils comme VirtualBox, vous instanciez des machines virtuelles sur votre PC.

Et finalement ce qu’on va voir c’est que la folie de la containerisation qu’à apporter Docker (et Kubernetes etc mais on en parlera pas ici… 👀), et bien c’est exactement pareil, ce sont des machines virtuelles. 🙂

C’est quoi un container ? 🧐#

Alors oui j’ai parlé de container, et il est bon d’expliquer le sens de ce mot. 😬

Un container est une machine virtuelle qui a pour but de porter un service précis pour répondre à un besoin.

Je pense que ça se passe de discours, l’idée du container c’est d’éviter d’instancier une grosse machine virtuelle pour porter tous les (micro) services d’un projet sous la même instance, et du coup éviter l’effet monolithique de la chose. 👍

Et Docker du coup ? 👀#

Et bien Docker c’est ce qu’on pourrait appeler un gestionnaire de container. 👍

Aujourd’hui Docker rassemble plusieurs services dans un package complet, on peux le dissocier de la façon suivante :

Le Docker engine#

C’est le moteur de docker, c’est le “backend” qui va construire les images des containers, les instancier, brancher les volumes, faire le routing réseaux etc…

En gros c’est le coeur de Docker, il est toujours accompagné et drivé par le CLI docker qui permet de le contrôler. 💪

Docker Desktop#

Sur votre PC, vous pourrez installer Docker Desktop et ainsi disposer d’une interface pour piloter et configurer le Docker Engine.

Attention, selon l’usage de vos containers, une licence peut être nécessaire, surtout si vous bossez dans un contexte d’entreprise, même si dans les faits personne n’ira vérifier. 👀

Installation de Docker (et Compose)#

Historique#

Il fût un temps où Docker et Docker Compose ne faisait pas partie du même package d’installation, on disposait d’un CLI docker et un CLI docker-compose séparés.

Aujourd’hui les contextes de déploiement sont si souvent multi-service que Docker a choisi d’intégrer Compose comme partie intégrante de Docker à l’installation, ce qui fait que compose est maintenant une commande du CLI docker et non plus un CLI à part.

Installation#

Pour l’installation, rien de compliqué, je vous invite à suivre ce lien et la documentation associé.

On peux résumer l’installation en deux déclinaisons :

  • La version “Desktop”, qui vous permet d’installer un package complet avec l’interface pour votre PC. 💻
  • La version “Server”, qui vous permet d’installer la version CLI pour un serveur qui ne dispose pas d’environnement de bureau. 🛰️

Si vous venez de mon article sur Matomo , prenez le temps d’installer dans un premier temps la version Desktop en local pour tester tout ce qu’on va voir ici et vous aurez ensuite tout le temps de préparer votre serveur.

Premier démarrage#

Pour vérifier que tout est ok je vous invite à ouvrir un terminal (PowerShell ou Git Bash ou ce que vous voulez, peu importe) et taper docker ps, et docker compose, si ces deux commandes ne renvoient pas d’erreurs, c’est que tout va bien. 🙂

(Sur windows, selon les versions il faut parfois utiliser docker-compose à l’ancienne, et non docker compose, dans la pratique ça ne change rien à part un tiret).

Comment ça marche#

Ok alors maintenant, parcourons les bases de Docker ensemble. 🧠

Maaaaiiss non vous en faites pas, aller, on y va. 😎

La commande docker ps vous permet de lister les containers en train de tourner, docker ps -a vous affichera également ceux qui ont été “terminés” (fermés) récemment.

Prenons un exemple concret grâce au Hello World, lancez cette commande : docker run hello-world.

Ce container lancé, il va vous expliquer ce qu’il s’est passé mais je vous le réexplique quand même ici en VF 😁 :

  • Docker va tirer l’image hello-world du Docker Hub.
  • Il va créer un nouveau container qui va instancier cette image et lancer les commandes de son Dockerfile.
  • Docker va vous afficher la sortie de la console du container à l’écran.

Le vocabulaire de Docker#

Si vous êtes un peu perdu, c’est normal, j’ai mis en gras les mots clés que je vous explique tout de suite. 🙂

Un Container 🚀#

Voilà ce qui s’apparente le plus à la machine virtuelle, un container est une instanciation d’une image Docker, sur laquelle vous allez (éventuellement) pouvoir vous connecter et qui va héberger votre (ou une partie de votre) projet. 👍

Vous pouvez voir la liste des container avec docker ps, le paramètre -a pernet de lister ceux qui sont à l’arrêt également.

Une image 🖼#

L’image Docker est un peu le fichier d’une VM à l’arrêt, c’est ce qui va être instancié par Docker pour lancer un container, c’est ce qui résulte du build d’un Dockerfile. 💪

Vous pouvez lister les images en tapant docker images dans votre terminal.

Dockerfile 🗄#

Ce fichier représente la recette d’une image, c’est un fichier qui permet de construire une image, souvent en partant d’une image existante et en lançant des commandes comme on le ferais sur notre terminal. 🤓

Toute image Docker est basée sur un Dockerfile. 👍

DockerHub 🌐#

L’équivalent de GitHub pour les images Docker, il permet de centraliser les images Docker, de la même manière que GitHub centralise des repositories Git, tout simplement. 👍

C’est ce qui fait que quand on indique le nom d’une image dans notre commande docker ou dans un docker-compose.yml, Docker retrouve tout seul le Dockerfile de l’image recherchée et le télécharge, c’est parce qu’il est branché au DockerHub et va aller voir si l’image indiquée existe sur DockerHub.

(Et oui, il y a abus de langage entre une image et le Dockerfile d’une image, c’est simplement qu’une image est le résultat d’un build de Dockerfile, donc ces deux termes sont intimement liés ). 😉

Volume 💾#

Oui, celui-ci n’y est pas mais je vous l’explique quand même. 😄

Parfois il est nécessaire de persister des données sur le serveur ou votre machine, par exemple si vous instanciez un container de base de donnée, vous n’avez pas envie que vos données soient perdues si votre container crash ou quand vous redémarrez. 😕

Et bien pour ce faire vous pourrez configurer un Volume qui va simplement “brancher” un dossier de votre container en local, vous pouvez le brancher à un dossier de votre machine pour pouvoir le voir et interagir avec, ou simplement le déclarer à Docker pour qu’il persiste les données sans vous les présenter parmi vos fichiers, dans les deux cas si vous redémarrez votre container ou reconstruisez votre image, vous récupérerez les données du volume créé précédemment. 📁

La commande docker volume ls vous permet de lister les volumes de votre machine 👍

$ docker volume ls                                                                    DRIVER    VOLUME NAME                                                                   local     nom-du-volume1
local     nom-du-volume2
local     nom-du-volume3

(Deux volumes ne peuvent porter le même nom)

Voilà, avec tout ça vous avez le lexique important de Docker, et toute l’articulation qui en résulte. 😄

(Si vous avez envie de jeter votre cerveau dans l’immédiat, c’est normal, n’hésitez pas à faire une pause pour trifouiller tout ce qu’on vient de voir.) 😬

Utilisation de Docker compose ✔#

Compose est un orchestrateur de container Docker, ça veut dire qu’il se base sur un fichier docker-compose.yml pour instancier des containers de services selon la configuration donnée.

Je ne vais pas vous repeindre la documentation de docker-compose mais simplement vous donner un exemple tiré de l’article sur Matomo (vous me dites si je force hein. 👀).

version: "3"

services:
  db:
    image: mariadb:10.11
    command: --max-allowed-packet=64MB
    restart: always
    volumes:
      - db:/var/lib/mysql:Z
    environment:
      - MYSQL_ROOT_PASSWORD=
      - MARIADB_AUTO_UPGRADE=1
      - MARIADB_DISABLE_UPGRADE_BACKUP=1
    env_file:
      - ./db.env

  app:
    image: matomo:fpm-alpine
    restart: always
    links:
      - db
    volumes:
      # - ./config:/var/www/html/config:z
      # - ./logs:/var/www/html/logs:z
      - matomo:/var/www/html:z
    environment:
      - MATOMO_DATABASE_HOST=db
      - PHP_MEMORY_LIMIT=2048M
    env_file:
      - ./db.env

  web:
    build:
      context: ./nginx
    restart: always
    volumes:
      - matomo:/var/www/html:z,ro
      # see https://github.com/matomo-org/matomo-nginx
      - ./nginx/matomo.conf:/etc/nginx/conf.d/default.conf:z,ro
      - letsencrypt:/etc/letsencrypt
      - ./nginx/certbot.sh:/etc/certbot.sh
    ports:
      - 80:80
      - 443:443
    depends_on:
      - app
    links:
      - app


volumes:
  db:
  matomo:
  letsencrypt:

Donc si on lit ce fichier on peut voir qu’on a trois services : db, app, et web.

  • 1‍⃣ db héberge notre base de donnée, on vois qu’il se base sur mariadb, et qu’il prend quelques variables d’environnement pour se construire, on vois également qu’il dispose d’un volume NOM DU VOLUME pour persister ses données.
  • 2‍⃣ Ensuite on a app, là c’est une image de matomo, vous pouvez la retrouver ici sur Docker Hub pour en savoir plus, elle permet d’installer et démarrer une instance Matomo, on vois qu’elle pointe sur le port 9000 et est reliée (en termes de réseau) à db, ce links permet d’établir des virtual host et ainsi assurer la communication entre les containers sur le réseau.
  • 3‍⃣ Et enfin, web est ce qui va gérer l’exposition sur internet, elle se base sur nginx et se sert d’un fichier de config “branché” via un volume pour tourner, on vois que le service expose son port 80 et qu’il est relié à l’app.

Donc quand on lance docker-compose up, par défaut docker va chercher le fichier docker-compose.yml dans le dossier courant et faire sa compote, rien de magique ou de caché. 🙂

Eh beh oui, c’est tout, et vous verrez qu’avec un peu de prise en main, tout ce fonctionnement vous semblera assez naturel. 🙂

Conclusion#

Je pense qu’on a fait le tour, avec ça vous avez les grandes base de Docker et pourrez commencer à l’utiliser de manière concrète en comprenant ce qui se trame derrière. 😉
